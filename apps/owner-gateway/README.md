# TemichevVet Owner Gateway

Отдельный публичный шлюз личного кабинета владельца. Он не подключается к базе CRM и не открывает локальный API клиники в интернет.

## Граница данных

CRM сама исходящим запросом передаёт только разрешённый снимок одного владельца:

- контактные данные самого владельца;
- его пациентов, последние веса и прививки;
- записи без внутренних комментариев;
- только завершённые приёмы;
- только подписанные документы;
- счета;
- только отправленные внешние уведомления без технического адресата.

Шлюз хранит этот снимок в собственной PostgreSQL-базе. Схема и миграция из `apps/owner-gateway/prisma` относятся только к этой отдельной базе и никогда не должны запускаться с `DATABASE_URL` клиники.

## Поток приглашения

1. Сотрудник выбирает MAX, Telegram или ссылку/QR в карточке владельца.
2. CRM создаёт одноразовый токен на 24 часа и синхронизирует разрешённый снимок через `X-Owner-Gateway-Secret`.
3. Для MAX или Telegram ссылка запускает бота с токеном. Публичный webhook связывает пользователя с владельцем и возвращает кнопку кабинета.
4. При активации токен погашается ровно один раз, а браузер получает `HttpOnly` cookie сессии.
5. Кнопка «Обновить данные кабинета» в CRM повторно отправляет разрешённый снимок без нового приглашения.

## Сообщения владельцу

CRM хранит очередь и время отправки локально. Когда наступает срок, она исходящим запросом вызывает защищённый `POST /internal/v1/owners/:ownerId/messages`. Шлюз выбирает только реально привязанный MAX или Telegram, отправляет сообщение соответствующим ботом и не открывает локальную CRM для входящих запросов.

Для Telegram требуются `TELEGRAM_BOT_USERNAME`, `TELEGRAM_BOT_TOKEN` и `TELEGRAM_WEBHOOK_SECRET`. Webhook бота должен указывать на `https://cabinet.temichevvet.ru/v1/webhooks/telegram` и передавать тот же секрет в заголовке `X-Telegram-Bot-Api-Secret-Token`.

## Push в установленном кабинете

Web Push хранит только подписку конкретного устройства в отдельной базе шлюза. Подписка связана с активной сессией кабинета и перестаёт использоваться после выхода или сброса входов. Для отправки нужны один раз созданные VAPID-ключи в `OWNER_GATEWAY_VAPID_PUBLIC_KEY` и `OWNER_GATEWAY_VAPID_PRIVATE_KEY`, а также контакт `OWNER_GATEWAY_VAPID_SUBJECT`.

Миграция `20260724000200_portal_web_push` создаёт только таблицу `PortalPushSubscription` в базе owner-gateway. К базе клиники эта миграция не применяется.

## До отдельного решения о деплое

- не настраивать MAX webhook и не публиковать порт CRM;
- оставить `OWNER_GATEWAY_URL` и `OWNER_GATEWAY_SYNC_SECRET` пустыми в локальной CRM;
- не запускать миграцию шлюза против базы клиники;
- онлайн-запись остаётся выключенной.

Локальная проверка сборки, не запускающая БД или сервер:

```bash
npm --workspace @clinic-crm/owner-gateway run build
```
