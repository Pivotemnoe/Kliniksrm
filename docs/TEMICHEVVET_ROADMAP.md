# Дорожная карта TemichevVet

Цель: сделать TemichevVet лучшей версией Vetaf под работу клиники, но на нашем стеке: React, Vite, TypeScript, Ant Design, TanStack Query, NestJS, Prisma, PostgreSQL, Redis, Docker Compose, OpenAPI.

## Принцип разработки

- Сначала повторяем рабочую логику Vetaf, чтобы система была понятна сотрудникам.
- Потом улучшаем сценарии под клинику Темичева.
- Все действия в системе должны быть от имени сотрудника.
- Директор управляет сотрудниками, ролями и точечными правами.
- Врач работает только под своим аккаунтом.
- Медицинские данные не хранятся в браузере как источник истины, всё идёт из backend и PostgreSQL.
- Frontend показывает права, но главным источником проверки всегда остаётся backend.

## Уже заложено

- Авторизация через HttpOnly cookie-session.
- Автовыход врача/сотрудника после 15 минут простоя: frontend завершает сессию, backend хранит такой же idle timeout и требует повторный вход.
- Роли и базовые права.
- Владельцы и пациенты.
- Каталог видов и пород.
- Очередь и отдельный экран очереди для клиентов.
- Записи на приём.
- Приёмы, лист осмотра, диагнозы, рекомендации, услуги.
- Документы приёма: шаблоны, создание документа из шаблона, редактирование статуса и текста.
- Счета и оплаты на backend.
- Баланс владельца, операции депозита, оплата счёта с депозита и возврат на депозит.
- Склад на backend и базовый frontend.
- Стационар на backend и базовый frontend.
- Сотрудники.
- Audit log с экраном просмотра последних действий сотрудников.
- Docker Compose, PostgreSQL, Redis, Prisma.
- Полный локальный backup: PostgreSQL, Redis и MinIO.
- Безопасное локальное восстановление из backup с явным подтверждением.
- Production web-контейнер: собранный React frontend через Nginx с прокси `/api`.

## Этап 1. Vetaf-подобный рабочий интерфейс

Статус: в работе.

Что делаем:

- Левое меню как дерево Vetaf: новости, сводка, расписание, очередь, задачи, владельцы, пациенты, приёмы, счета, продажи, стационар, склад, настройки.
- Раздел новостей заменён на рабочую внутреннюю ленту: публикации по ролям, важность, закрепление, архив и отметка прочтения сотрудником.
- Верхняя панель: поиск, сообщения, онлайн-запись, уведомления, финансы, профиль сотрудника.
- Верхний поиск стал быстрым поиском по владельцам и пациентам с переходом в карточку.
- Колокольчик верхней панели показывает непрочитанные новости, реальные ошибки отправки сообщений и новые онлайн-заявки, переход ведёт в доступный сотруднику раздел.

- Рабочие списки с плотными таблицами.
- Отдельные страницы под будущие Vetaf-модули, чтобы структура проекта больше не расползалась.

## Этап 13. Локальный сервер и автономная работа

Статус: частично готово.

Принцип: CRM должна работать внутри клиники на локальном сервере даже при потере интернета. Интернет нужен только для внешних интеграций: Telegram, MAX, SMS, email, онлайн-запись, обновления и удалённый доступ.

Что нужно:

- Установка на компьютер или мини-сервер клиники через Docker Compose.
- Локальная сеть: рабочие места открывают CRM по адресу сервера внутри клиники.
- PostgreSQL, Redis, MinIO, backend и web работают локально.
- Очередь исходящих уведомлений сохраняется локально и отправляется после восстановления интернета.
- Статусы доставки и ошибки видны администратору.
- Резервное копирование базы и файлов на внешний диск или сетевое хранилище.
- Опциональный внешний домен/VPN для удалённого доступа директора и личного кабинета клиента.
- Для PWA-кабинета клиента нужен HTTPS-доступ с телефона владельца: локальная CRM может оставаться внутри клиники, а наружу отдаётся только ограниченный публичный портал через домен, reverse proxy/VPN/tunnel или отдельный небольшой шлюз.
- Безопасный режим: если интернет пропал, приём, очередь, склад, счета и стационар продолжают работать.

Что уже добавлено:

- Docker Compose поднимает PostgreSQL, Redis, MinIO и backend локально.
- Docker Compose поднимает web-контейнер с Nginx: frontend отдаётся локально, `/api` проксируется к backend.
- Сессии, очередь, приёмы, счета, склад и стационар работают через локальные сервисы.
- `npm run backup:local` создаёт единый архив с PostgreSQL dump, Redis snapshot и данными MinIO.
- `npm run backup:postgres` оставлен для быстрого отдельного дампа базы.
- `npm run restore:local -- <архив>` проверяет архив, а `CONFIRM_RESTORE=YES` выполняет восстановление PostgreSQL и MinIO.
